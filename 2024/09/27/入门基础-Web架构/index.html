网安入门-Web架构

发表于2024-09-27|更新于2024-09-27|网安

Web架构

网站常见搭建模式

常规的Web应用搭建步骤：

1、购买云服务器，购买域名

2、云服务器去搭建中间件

3、下载并上传Web程序源码

4、添加网站并绑定域名目录

子域名模式：

www.example.com //zblog asp程序

example.example.com //wordpress php程序

bbs.example.com //论坛 dz程序 BBS :社区论坛 dz :Discuz!软件搭建的论坛

oa.example.com //OA 通达OA OA（Office Automation）办公自动化

端口模式：

www.example.com //zblog asp程序

www.example.com:8080 //wordpress php程序

相当于两个目标 程序是我们攻击的目标

目录模式：

www.example.com //zblog asp程序

www.example.com/bbs //论坛 dz程序

Docker模式：

Docker容器技术来部署、管理和扩展网站，安全性高

容器化（Containerization）：将应用程序及其依赖环境打包在一起，形成一个可移植的容器

镜像（Image）：容器的静态模板，用于创建容器

容器（Container）：镜像的运行实例

分配给用户使用

Web程序源码分类与结构：

开源：

源码可见

源码不可见 加密（加密工具）或语言特性决定（JAVA）

商业(闭源)：网校系统 开源的功能不行 花钱开发

自写：大型企业 政要单位

对目标判断拿到源码的代价及可能性

目录结构Admin database bin

对中间件的配置：

攻击者角度：植入后门（木马）–连接控制网站

管理员对中间件的设置阻止你：

1，设置目录的属性-安全，权限拒绝后，此手段失效，但是一般只能锁一些图片目录（锁了还能正常读取），不然网站都打不开

2，目录-身份验证-禁用匿名身份验证-启用Windows身份验证，访问网站需要账号密码才能访问

站库分离

Access MSSQL 前者不用账号密码本地建库，后者要，可以站库分离

源码里有数据库配置相关的信息

数据库与源码分离，两台服务器

以前建数据库还要自己有个操作系统，现在有云数据库

现在你拿下了这个网站，数据库在本地也能拿到，数据库在另一台服务器不一定能拿到，数据库在云，几乎不可能

以前上传木马，拿到网站，现在云对象存储OSS，上传了没有卵用

常规访问和路由访问

常规访问：URL和文件目录对应

路由访问：URL和文件目录对不上，mvc源码 JAVA Python，要根据路由配置决定

前后端分离

原理：前端JS框架，API传输数据

理解：前端一个页面，后端一个页面，一般页面十分简洁美化没有那么多的分类

一般没分离的可以/admin之类的访问

影响：

1、前端页面大部分不存在漏洞

2、后端管理大部分不在同域名

3、获得权限有可能不影响后端（是否同域，子域名）

软件搭建：宝塔或Phpstudy

原理：打包类集成化环境，权限配置或受控制

影响：攻击者权限对比区别

宝塔：攻击者上传连接后，文件被锁定目录，命令无法执行

自己搭建的IIS：部分可看部分可执行 iis apppool\defaultapppool

Phpstudy（创建时安全可设置，默认全关）：文件目录可看，命令可执行 whomi 获取用户权限—administrator

Docker容器

原理：虚拟化技术独立磁盘空间，非真实物理环境

影响：攻击者虚拟空间磁盘

判断是否分配给用户使用

虚构一个了一个假象给你，目录不一致

你就像docker容器搭建的网站，当我好不容易得到你的芳心（权限），想再靠近你，触摸你，可是举目四望，一切竟只是假象。

建站分配站

1.托管 拥有版权

2.申请 没有版权

原理：利用别人域名模版建立

影响：实质安全测试非目标资产

你测试对象实际是平台不是用户

静态网站

例子：大学学的html设计的网站

原理：数据没有传输性（js传输不算）

影响：无漏洞

伪静态网站：

动态转为静态技术，伪装的静态

动态网站：

内容动态生成，用户交互，实时更新

WAF（防火墙防护）

原理：Web应用防火墙，旨在提供保护

影响：常规Web安全测试手段会受到拦截

演示：免费D盾防护软件

Windows2012 + IIS +D盾

CDN（内容分发网络）

原理：内容分发服务，旨在提高访问速度

影响：隐藏真实源IP，导致对目标测试错误

演示：阿里云备案域名全局CDN加速服务

Windows2012 + BT宝塔面板 + CDN服务

超级ping网站：https://ping.chinaz.com来判断是否使用了CDN

OSS（云存储）

原理：云存储服务，旨在提高访问速度

影响：

演示：https://cloudreve.org/

Windows2012 + cloudreve + 阿里云OSS

https://github.com/cloudreve/Cloudreve/releases/tag/3.7.1

1、启动应用

2、登录管理

3、配置存储信息

4、更改用户组存储属性

阿里云OSS:

1、开通OSS

2、新建Bucket

3、配置Bucket属性

4、配置Access访问

原理：

为什么要使用第三方存储？

1）静态文件会占用大量带宽

2）加载速度

3）存储空间

影响：

上传的文件或解析的文件均来自于OSS资源，无法解析，单独存储

1、修复上传安全,上传了，但传到了另一个无法解析的地方

2、文件解析不一样

3、但Accesskey隐患，如果我能获得OSS的密码

反向代理

正向代理为客户端服务,客户端主动建立代理访问目标（不代理不可达）

所以正向没有影响

反向代理为服务端服务,服务端主动转发数据给可访问地址（不主动不可达）

原理：通过网络反向代理转发真实服务达到访问目的

影响：访问目标只是一个代理，非真实应用服务器

注意：正向代理和反向代理都是解决访问不可达的问题，但由于反向代理中多出一个可以重定向解析的功能操作，导致反代理出的站点指向和真实应用毫无关系！

负载均衡

原理：分摊到多个操作单元上进行执行，共同完成工作任务

影响：有多个服务器加载服务，测试过程中存在多个目标情况

演示：Nginx负载均衡配置

Windows2012 + BT宝塔面板 + Nginx

#定义负载设置

upstream fzjh{

​ server 47.94.236.117:80 weight=2; #权重更高，优先级更高，打开概率更高

​ server 47.122.22.195:80 weight=1;

} #用这两个网站提供服务

#定义访问路径 访问策略

location / {

proxy_pass http://fzjh/;

}#访问这个根目录时启动负载均衡
